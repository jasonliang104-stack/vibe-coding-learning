# 简历优化微信小程序 技术选型与系统方案初稿

版本：`V1.0`
更新时间：`2026-04-21`
适用范围：基于《简历优化微信小程序 PRD》的第一版技术方案

---

## 1. 文档目的

本文档用于沉淀第一版简历优化微信小程序的技术选型、系统架构、模块拆分和关键实现策略，作为后续开发、排期、数据库设计、接口设计和部署方案的基础依据。

本文档聚焦：

- 第一版小程序主链路落地
- 适合个人开发和 Codex 协作的技术方案
- 保证后续可扩展，但不过度设计

---

## 2. 技术选型总览

### 2.1 前端

- 前端框架：`微信原生小程序`
- 开发语言：`TypeScript`
- UI 策略：`尽量原生 + 少量自定义组件`
- 样式方案：`原生 WXSS + 少量公共样式`
- 页面组织：`主包 + 分包`
- 目录组织：`按业务域分目录`
- 状态管理：`轻量方案，页面各自管理状态 + 少量公共状态封装`
- 网络请求：`基于 wx.request 做统一封装`
- 页面数据获取：`页面进入时按需拉取数据`
- 页面间数据传递：`只传必要参数，详情数据到目标页再拉取`
- 本地缓存：`只缓存少量必要信息`
- 上传组件：`抽一个轻量统一上传组件，再按场景配置`
- 表单处理：`原生 data / setData 管理表单`
- 错误提示：`按场景分层提示（toast + 弹窗 + 页面态）`
- 导航实现：`第一版优先使用微信原生 navigationBar + 原生 tabBar`
- 埋点策略：`第一版做基础埋点`

### 2.2 主业务后端

- 技术栈：`Node.js + NestJS`
- 接口风格：`RESTful API`
- 接口命名：`REST 为主，少量任务型接口按业务动作命名`
- 鉴权方式：`JWT / Token 鉴权`
- 用户主键：`业务侧 user_id，openid 作为微信登录绑定字段`
- 数据访问：`NestJS + ORM`
- ORM：`Prisma`
- 参数校验：`框架层统一参数校验`
- 返回格式：`统一返回结构`
- 接口文档：`自动生成接口文档`
- 代码分层：`控制器 + 服务 + 数据访问层 的基础分层`
- 数据库迁移：`使用正式迁移机制管理变更`
- 错误处理：`建立统一错误码体系`
- 日志策略：`第一版先做基础日志`

### 2.3 AI / 文档服务

- 技术栈：`Python + FastAPI`
- 服务职责：`AI 能力单独拆一个服务`
- 服务间通信：`HTTP API 调用`
- 内部编排：`按步骤拆成可组合的处理链`
- Prompt 管理：`独立 Prompt 管理层`
- AI 结果处理：`模型输出后做结果校验和清洗`
- 导出文件生成：`放在 AI / 文档服务中处理`
- 文档处理：`先用开源方案`

### 2.4 数据与基础设施

- 数据库：`MySQL`
- 缓存：`Redis`
- 文件存储：`腾讯云 COS`
- 支付：`微信支付`
- 部署体系：`腾讯云体系`
- 部署方式：`第一版先用云服务器部署`
- 对象存储访问：`文件默认私有，通过后端生成临时访问/下载地址`

### 2.5 任务与异步处理

- 任务机制：`异步任务 + 轮询状态`
- 实现方式：`数据库状态 + Redis + 前端轮询，第一版不引入正式消息队列`
- 轮询策略：`固定频率轮询`

## 3. 总体架构设计

### 3.1 仓库拆分策略

第一版采用前后端分仓，不做 monorepo。

建议拆分为 3 个仓库：

1. `miniapp-frontend`
   微信原生小程序前端仓库

2. `business-backend-nestjs`
   主业务后端仓库，负责用户、任务、积分、会员、订单、文件元数据等业务能力

3. `ai-service-fastapi`
   AI / 文档服务仓库，负责 JD 提取、简历诊断、简历优化、文档导出等能力

### 3.2 服务拆分策略

第一版不做微服务化扩展，保持：

- `1 个主业务服务`
- `1 个 AI / 文档服务`

目标是保证开发效率和联调效率，避免过度拆分导致复杂度过高。

## 4. 前端技术方案

### 4.1 小程序页面分包策略

#### 主包页面

- 首页
- 简历列表
- 我的

说明：

- 启动页和 `tabBar` 页面放主包
- 保持主包尽量轻

#### 分包一：`pkg-optimize`

用于核心优化流程：

- 简历优化提交页
- 诊断动画页
- 诊断报告页
- 补充信息页
- 简历优化中过渡页
- 优化结果确认页
- 格式对齐页

#### 分包二：`pkg-resume`

用于简历和任务详情相关页面：

- 任务详情页
- 原始简历查看页
- 简历查看页

#### 分包三：`pkg-account`

用于账户与付费相关页面：

- 会员/积分充值页
- 订单记录页
- 积分明细页
- 问题反馈页
- 视频奖励结果页（如单独成页）

### 4.2 视频广告相关页面策略

- 优先做成弹层/组件，不单独拆页面
- 如果需要独立页面，则放在 `pkg-account`

### 4.3 前端目录建议

建议目录结构按业务域组织：

- `pages/`：主包页面
- `pkg-optimize/`：优化流程分包
- `pkg-resume/`：简历与任务详情分包
- `pkg-account/`：账户与付费分包
- `components/`：公共组件
- `services/`：接口请求封装
- `store/`：少量全局状态
- `utils/`：工具函数
- `constants/`：常量
- `types/`：类型定义

### 4.4 前端关键实现策略

#### 状态管理

- 页面内状态由页面自身管理
- 用户登录态、会员状态、积分余额等少量数据做公共状态封装

#### 请求封装

- 基于 `wx.request` 封装统一请求层
- 统一处理 token、错误提示、超时、返回结构

#### 文件上传

- 前端上传到后端
- 后端校验并统一上传 COS
- 不采用前端直传 COS

#### 下载策略

- 后端生成下载地址
- 前端调用下载能力

#### 错误提示体系

- 轻提示：`toast`
- 需用户决策：`弹窗`
- 页面级状态：`空态 / 错误页 / 任务失败态`

## 5. 主业务后端技术方案

### 5.1 模块划分

NestJS 主业务服务按模块组织，不继续拆成微服务。

建议模块如下：

- `auth`：微信登录、token、身份校验
- `user`：用户资料、账户基础信息
- `resume`：原始简历、优化后简历、简历列表
- `jd`：JD 输入记录、截图文件信息、JD 结构化结果引用
- `task`：诊断任务、优化任务、任务状态、轮询查询
- `points`：积分余额、积分流水、扣减/返还
- `reward`：激励视频广告奖励、奖励积分发放、奖励事件记录、幂等控制
- `membership`：会员状态、套餐权益、赠送积分
- `payment`：微信支付、订单、支付回调
- `file`：文件元数据、COS 路径、下载地址生成
- `model`：模型配置、模型展示信息、速度优先/质量优先映射
- `analytics`：埋点事件记录
- `common`：错误码、统一响应、配置、日志、通用工具

### 5.2 数据访问策略

- 采用 `Prisma` 作为 ORM
- 表结构变更通过正式迁移机制管理
- 日常查询以 ORM 为主
- 少量复杂查询可补充手写 SQL

### 5.3 鉴权与用户体系

- 登录方式：仅微信登录
- 用户主键：内部 `user_id`
- 微信身份：保存 `openid`
- 鉴权方案：`JWT / Token`

### 5.4 接口设计原则

- 统一返回结构
- 接口带版本号，例如 `/api/v1/...`
- 常规资源接口按 REST 风格设计
- 少量动作型接口允许按业务动作命名

### 5.5 资产与计费逻辑

#### 积分体系

- 采用 `余额 + 积分流水明细`
- 积分扣减与增加均写流水

#### 积分扣减机制

- 采用 `幂等控制 + 事务处理`
- 产品规则上支持“先扣积分，再创建任务”
- 但技术实现必须补偿：
  - 扣分成功但任务创建失败时返还积分
  - 全链路记录流水与失败原因

#### 支付处理

- 统一使用微信支付
- 支付结果以 `后端支付回调` 为准

#### 激励视频奖励

- 前后端都显式支持视频奖励链路
- 后端负责：
  - 奖励发放校验
  - 幂等控制
  - 次数限制
  - 奖励事件记录

## 6. AI / 文档服务技术方案

### 6.1 模块划分

FastAPI AI / 文档服务建议按以下模块组织：

- `jd_extractor`
  - 接收 JD 截图或文本
  - 调用大模型完成内容提取、整理与结构化

- `resume_parser`
  - 解析 Word 简历
  - 提取结构化简历信息

- `diagnosis`
  - 简历诊断
  - 匹配度分析
  - 优化建议生成

- `supplement`
  - 补充信息问题生成

- `rewrite`
  - 简历改写与优化

- `document`
  - Word 生成
  - 格式对齐
  - PDF 导出

- `model_gateway`
  - 模型适配层
  - 统一模型调用
  - 模型元信息管理

- `prompt`
  - Prompt 模板
  - Prompt 编排

- `common`
  - 配置、日志、错误处理、工具函数

### 6.2 模型接入策略

- 模型统一通过 `model_gateway` 接入
- 不直接在业务代码中写死具体厂商 SDK
- 页面固定展示 2 个模型选项：
  - `速度优先`
  - `质量优先`

### 6.3 模型记录与结果处理

#### 模型调用记录

保存以下内容：

- 输入内容引用或摘要
- 模型信息
- 输出结果
- 调用状态
- 失败原因
- 调用元数据

#### 输出结果处理

- 模型输出后做结果校验和清洗
- 不直接把原始模型输出透传到前端或导出链路

### 6.4 编排方式

AI 链路采用可组合处理链，而非单一大函数。

示例链路：

1. JD 提取
2. 简历解析
3. 简历诊断
4. 补充信息生成
5. 简历改写
6. 文档生成

## 7. 文件与存储策略

### 7.1 文件上传与下载

- 上传：前端传到后端，后端上传 COS
- 下载：后端生成临时下载地址，前端下载

### 7.2 文件存储组织

按以下维度分层组织：

- 用户
- 任务
- 文件类型

### 7.3 文件生命周期

#### 长期保存

- 原始简历
- 优化后简历
- 导出文件

#### 按规则清理

- JD 截图：按 90 天规则保留
- 临时处理中间文件：短期保留后清理
- 已逻辑删除文件：定时任务清理

### 7.4 删除策略

- 业务上采用逻辑删除
- 物理文件后续统一清理

### 7.5 存储访问权限

- 文件默认私有
- 通过后端生成临时签名地址访问

## 8. 任务与异步处理方案

### 8.1 异步任务机制

项目中的诊断和优化流程都采用异步任务机制：

- 创建任务
- 更新任务状态
- 前端固定频率轮询
- 完成后自动进入下一页

### 8.2 状态管理策略

第一版采用：

- 数据库存任务记录
- Redis 缓存任务状态
- 前端轮询获取最新状态

不引入正式消息队列。

### 8.3 任务数据保存

保存：

- 原始输入
- 中间结果
- 最终结果
- 任务状态变更记录

### 8.4 结果版本化

保留结果版本历史，包括：

- 原始版本
- 诊断结果版本
- 优化结果版本
- 格式对齐版本
- 更换模型重跑版本

## 9. 环境、部署与配置管理

### 9.1 环境策略

第一版采用：

- 开发环境
- 生产环境

测试环境后续补充。

### 9.2 配置管理

- 使用环境变量和配置文件
- 敏感信息不写入代码和仓库

### 9.3 部署方式

第一版采用腾讯云云服务器部署。

建议部署结构：

- Nginx
- NestJS 主业务服务
- FastAPI AI / 文档服务
- MySQL
- Redis
- 腾讯云 COS

### 9.4 定时任务能力

第一版预留基础定时任务能力，用于：

- 清理过期 JD
- 清理逻辑删除文件
- 清理临时文件
- 异常任务补偿

## 10. 安全、隐私与稳定性策略

### 10.1 隐私与安全

第一版按较高标准处理，但优先落地核心必要项。

至少做到：

- 只收集必要信息
- 传输全程 HTTPS
- 文件和敏感数据严格访问控制
- 用户只能访问自己的数据
- 支持用户删除数据
- 日志避免打印敏感正文

### 10.2 错误处理

建立统一错误码体系，覆盖：

- 登录类
- 文件类
- 任务类
- 模型调用类
- 积分类
- 支付类
- 系统异常类

### 10.3 AI 调用失败策略

- 采用 `重试 + 失败兜底`
- 失败任务明确标记状态
- 按失败阶段和失败原因决定是否返还积分

### 10.4 备份恢复

第一版就做基础备份恢复策略：

- 数据库定期备份
- 保留最近备份副本
- 明确恢复流程

## 11. 资金与商业化实现边界

### 11.1 第一版优先级

资金和资产相关模块第一版先做最小闭环：

- 新注册用户体验积分
- 模型积分消耗
- 积分余额
- 积分流水
- 微信支付充值
- 会员开通
- 会员积分折扣
- 激励视频奖励积分
- 积分不足引导

复杂运营权益后补。

### 11.2 会员与积分相关核心原则

- 支付结果以后端回调为准
- 积分变动全部记流水
- 资产类操作必须考虑幂等
- 激励视频奖励要做次数限制和幂等控制

---

## 12. 第一版不优先做的内容

以下内容不作为第一版重点：

- 多服务微服务化拆分
- 正式消息队列体系
- 管理后台页面开发
- 复杂运营活动系统
- 复杂多环境完整体系
- 高度定制组件库
- 重型自动化测试体系

- 提升文档格式还原能力

## 14. 当前最终结论

本项目第一版的技术方案核心结论如下：

- 前端：`微信原生小程序 + TypeScript`
- 后端：`NestJS`
- AI 服务：`FastAPI`
- 数据库：`MySQL`
- 缓存：`Redis`
- 存储：`腾讯云 COS`
- 支付：`微信支付`
- 模型：`统一模型网关 + 2 个前端可选模型`
- 任务：`异步任务 + 固定轮询`
- 部署：`腾讯云云服务器`
- 仓库：`前后端分仓`
- 服务：`1 个主业务服务 + 1 个 AI/文档服务`

该方案的目标是：

- 保证第一版可落地
- 保证个人开发和 Codex 协作效率
- 避免过度设计
- 保留后续扩展空间
