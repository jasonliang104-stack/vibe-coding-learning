# 系统架构设计初稿

## 1. 文档目标

本文档用于定义“简历优化微信小程序”第一版的系统架构设计，明确前端、主业务后端、AI/文档服务、数据库、缓存、对象存储及外部能力之间的边界与协作方式，作为开发实现的架构基线。

## 2. 架构目标

- 支撑微信原生小程序第一版主链路快速落地
- 保持职责清晰，避免第一版过度微服务化
- 支撑长任务异步处理、积分扣减、会员与支付等核心业务
- 为后续后台配置、模型扩展、文档处理能力增强预留演进空间

## 3. 总体架构

系统采用“三层两服务”结构：

1. 展示层：微信原生小程序前端
2. 业务层：NestJS 主业务服务
3. AI/文档层：FastAPI AI/文档服务

底层依赖：

- MySQL：核心业务数据存储
- Redis：缓存、幂等控制、任务状态辅助
- 腾讯云 COS：文件存储
- 微信支付：会员与积分充值
- 大模型网关：统一对接速度优先 / 质量优先模型

## 5. 核心交互链路

### 5.1 诊断链路

1. 小程序提交 JD 与简历
2. 主业务服务校验参数并扣减积分
3. 主业务服务创建诊断任务
4. 主业务服务通过 HTTP 调用 AI 服务
5. AI 服务完成 JD 提取、简历解析、诊断与建议生成
6. AI 服务返回结果或写回结果
7. 主业务服务更新任务状态
8. 小程序轮询并展示诊断报告

### 5.2 优化链路

1. 用户补充优化要求或补充信息
2. 主业务服务创建优化任务
3. 主业务服务调用 AI 服务执行优化
4. AI 服务完成改写、文档生成
5. 提前生成可导出的 Word/PDF
6. 主业务服务保存版本记录、文件记录和下载记录
7. 小程序轮询完成后进入结果确认页

### 5.3 支付与积分链路

1. 用户发起会员开通或积分充值
2. 小程序向主业务服务创建订单
3. 小程序拉起微信支付
4. 微信支付回调主业务服务
5. 主业务服务校验支付结果并更新订单状态
6. 按业务规则发放积分或生效会员权益
7. 写入积分流水 / 支付流水 / 订单记录

### 5.4 视频奖励积分链路

1. 用户点击观看视频领积分
2. 小程序拉起激励视频
3. 视频完整播放后通知主业务服务
4. 主业务服务校验奖励事件是否合法
5. 发放积分并写入奖励事件与积分流水

## 6. 任务处理架构

第一版采用：

- 异步任务
- 前端固定频率轮询
- 数据库状态 + Redis 辅助

不引入正式消息队列。

原因：

- 第一版以快速落地为主
- 长任务数量有限
- 降低部署和调试复杂度

任务相关核心对象：

- `tasks`
- `task_steps`
- `task_inputs`
- `task_results`
- `task_result_versions`
- `model_call_logs`

## 7. 数据架构

### 7.1 MySQL

用于存储：

- 用户与登录绑定
- 简历与简历版本
- JD 记录与结构化结果
- 任务与任务步骤
- 模型调用记录
- 积分账户与流水
- 会员与套餐
- 订单与支付流水
- 文件与下载记录
- 埋点事件

### 7.2 Redis

用于：

- 任务状态辅助缓存
- 幂等控制
- 短期状态与频控
- 后续定时任务配合

### 7.3 COS

用于存储：

- 原始简历
- 优化后简历
- JD 截图
- 导出 Word/PDF
- 中间处理文件

文件策略：

- 默认私有
- 通过后端签发临时下载地址访问

## 8. 文件架构

文件按以下维度组织：

- 用户
- 任务
- 文件类型

建议分类：

- 原始简历文件
- 优化结果文件
- JD 截图文件
- 导出文件
- 临时处理文件

生命周期规则：

- 原始简历：长期保存
- 优化后简历：长期保存
- JD：保留 90 天
- 逻辑删除文件：后续定时清理物理文件

## 9. 安全与隐私架构

第一版按较高标准处理，但优先落地核心必要项：

- 全链路 HTTPS
- 文件私有存储
- 用户只能访问自己的资源
- 敏感配置通过环境变量管理
- 不在日志中打印大段简历正文和敏感信息
- 支持用户逻辑删除简历和历史记录

## 10. 部署架构

第一版部署策略：

- 腾讯云云服务器
- 不做复杂容器化编排

组件部署：

- 小程序前端：微信开发者工具构建与发布
- NestJS：云服务器部署
- FastAPI：云服务器部署
- MySQL：本地开发 + 云上正式实例
- Redis：本地开发 + 云上正式实例
- COS：腾讯云对象存储

环境策略：

- 开发环境
- 生产环境
- 测试环境后续补充

## 11. 演进方向

后续可逐步演进：

- 接入后台管理端
- 模型配置化与运营配置化
- 更完善的失败补偿与重试机制
- 更复杂的积分与活动体系
- 容器化部署
- 更完整的监控、告警与链路追踪

## 12. 当前第一版架构结论

第一版系统架构最终定为：

- 前端：微信原生小程序 + TypeScript
- 主业务后端：NestJS + Prisma + MySQL + Redis
- AI/文档服务：FastAPI
- 文件存储：腾讯云 COS
- 支付：微信支付
- 模型：统一模型网关 / 模型适配层
- 任务：异步任务 + 前端轮询
- 服务拆分：1 个小程序前端 + 1 个主业务服务 + 1 个 AI/文档服务

该架构满足第一版快速开发、边界清晰、可持续演进的目标。
